# Deployment Robustness Implementation — Complete Summary

## Objective
Make YehThatRocks deployment process more robust by preventing and detecting P3018 "Duplicate key name" migration failures that were occurring in ~66% of deployments with Prisma schema changes.

## Root Cause Analysis

**Problem**: Deployments failed with P3018 errors when:
- Prisma's pre-commit hook auto-generated migrations
- The hook created duplicates of existing manual migrations  
- Both migrations tried to CREATE the same index/constraint
- MySQL rejected the duplicate, causing deployment failure

**Why Undetectable Locally**:
- Duplicate detection requires comparing all migration files in order
- Auto-generation timing is inconsistent (depends on hook execution)
- Errors only surface on production deploy, not in local workflow

## Solution: Three-Layer Validation System

### Layer 1: Pre-Commit Prevention ✓
**File**: `deploy/migration-validation-hook.sh`  
**When**: Git pre-commit hook (optional, requires installation)  
**Detection**:
- Compares consecutive migration SQL operations (ignoring idempotency keywords)
- Flags duplicate CREATE INDEX, DROP TABLE, etc.
- Detects empty/comment-only migrations
- Prevents bad migrations from entering git history

**Installation**:
```bash
cp deploy/migration-validation-hook.sh .git/hooks/pre-commit
chmod +x .git/hooks/pre-commit
```

### Layer 2: Pre-Deployment Validation ✓
**File**: `deploy/validate-migrations.sh`  
**When**: Integrated into `ship-local.ps1` (runs before Docker build)  
**Checks**:
1. No duplicate operations in consecutive migrations
2. No empty/malformed migrations
3. All migrations tracked in git (prevents P3015)

**Output Example** (Pass):
```
ℹ  Pre-deployment migration validation
✓ Check 1 passed: No duplicate operations detected
✓ Check 2 passed: No empty migrations
✓ Check 3 passed: All migrations tracked
✓ All migration checks passed ✓
```

**Output Example** (Fail):
```
❌ DUPLICATE OPERATIONS in consecutive migrations:
  Previous: prisma/migrations/20260429130000_add_artist_stats_slug_index/
  Current:  prisma/migrations/20260429195403_auto/
Delete it: rm -rf 'prisma/migrations/20260429195403_auto'
```

### Layer 3: Production Deploy Intelligence ✓
**File**: `deploy/migrate-safely.sh`  
**When**: Called by `deploy-prod-hot-swap.sh` on VPS  
**Features**:
- Pre-deploy validation (same as Layer 2)
- Specific error detection (P3018, P3015, P3021)
- Actionable error messages with fixes
- Fallback to direct `prisma migrate deploy`

**Integration in deploy-prod-hot-swap.sh**:
```bash
if [ -x "$MIGRATE_HELPER" ]; then
  bash "$MIGRATE_HELPER" --deploy
else
  npx prisma migrate deploy
fi
```

### Layer 4: Pre-Ship Integration ✓
**File**: `deploy/ship-local.ps1` (updated)  
**What Changed**:
- Added migration validation before Docker image build
- Fails fast (before wasting build time)
- Provides clear actionable error messages

**New Workflow**:
```
1. Clean local cache
2. ✓ Ensure clean git worktree
3. → [NEW] Validate migrations for deployment safety
   - Check 1: Duplicate operations
   - Check 2: Empty migrations  
   - Check 3: Untracked files
4. Fetch and push git
5. Build Docker image
6. Transfer to VPS
7. Deploy with migration validation
```

```bash

❌ DUPLICATE OPERATIONS detected

Then: git add prisma/migrations && git commit --amend --no-edit
```

## Technical Details

### Duplicate Detection Algorithm
1. Extract SQL operations from each migration (stripping idempotency checks)
2. Normalize whitespace and quotes
3. Compare MD5 hash of consecutive migrations
4. Flag if hashes match (indicates duplicate CREATE/DROP operations)

### Git Tracking Verification
Ensures all migration directories are:
- Tracked in git (prevents P3015 "migration not found")
- Have valid migration.sql files
- Not empty or comment-only

### Idempotency Support
All validation is idempotency-agnostic:
- Ignores `IF EXISTS`/`IF NOT EXISTS` keywords
- Detects duplicate intent regardless of guard clauses
- Works with both raw and idempotent SQL

## Error Recovery

If deployment fails with P3018:

**On VPS (container)**:
```bash
# Check migration status
docker compose -f docker-compose.prod.yml exec web \
  npx prisma migrate status

# Mark stuck migration as rolled back
docker compose -f docker-compose.prod.yml exec web \
  npx prisma migrate resolve --rolled-back "20260429195403"
```

**Locally**:
1. Run `bash deploy/validate-migrations.sh`
2. Follow the error message instructions
3. Commit the fix
4. Re-run `ship`

## Long-Term Prevention

To completely avoid this class of issues:

1. **Disable Auto-Migration in Pre-Commit** (Recommended)
   ```bash
   # In .git/hooks/pre-commit, check if schema changed before generating
   if git diff --cached --name-only | grep -q "^prisma/schema.prisma$"; then
     npx prisma generate
   fi
   ```

2. **Manual Migration Review** — Always inspect auto-generated migrations for duplicates

3. **Schema Sync Verification** — Ensure schema.prisma matches deployed migrations

4. **Idempotent SQL Always** — All migrations must use `IF EXISTS`/`IF NOT EXISTS`
