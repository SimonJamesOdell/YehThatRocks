# YehThatRocks Deployment Robustness Improvements

## Summary

This document describes the multi-layer migration validation system implemented to prevent P3018 deployment failures caused by Prisma's inconsistent auto-generation of duplicate migrations.

## Problem Diagnosis

**Issue**: Approximately 66% of deployments with Prisma schema changes failed with P3018 ("Duplicate key name") errors.

**Root Cause**: Prisma's pre-commit hook sometimes auto-generates migrations that duplicate operations from manually-created migrations:
- Hook runs: `prisma generate` → `prisma migrate dev --create-only`
- If a manual migration already creates index `foo`, the hook creates `foo` again
- Both migrations end up in git, causing P3018 on deploy when MySQL rejects duplicate index creation

**Why It's Hard to Catch**: 
- The hook runs inconsistently depending on timing and git state
- Duplicate migrations only fail at deploy time on production, not locally
- Developers might not notice duplicates if they're in separate migration files with similar names

## Solution: Three-Layer Validation

### Layer 1 — Prevention (Git Commit Time)

**File**: `deploy/migration-validation-hook.sh`  
**When**: Automatically runs on `git commit` (if installed)  
**What it does**:
- Detects duplicate operations in consecutive migrations
- Checks for empty/malformed migrations
- Prevents bad migrations from entering git history

**Installation**:
```bash
cp deploy/migration-validation-hook.sh .git/hooks/pre-commit
chmod +x .git/hooks/pre-commit
```

**Example: Catches**
```
❌ DUPLICATE OPERATIONS in consecutive migrations:
  Previous: prisma/migrations/20260429130000_add_artist_stats_slug_index/
  Current:  prisma/migrations/20260429195403_auto/

The current one likely auto-generated by Prisma hook.
```

### Layer 2 — Validation (Pre-Deploy Time)

**File**: `deploy/validate-migrations.sh`  
**When**: Runs before deployment via `bash deploy/validate-migrations.sh`  
**What it does**:
- Scan migrations for duplicate CREATE/DROP operations
- Check for untracked migration files (causes P3015)
- Provide actionable fix commands

**Usage**:
```bash
# Run manually before deploying
bash deploy/validate-migrations.sh

# Or in CI/pre-deployment pipeline
bash deploy/validate-migrations.sh /path/to/repo
```

**Output example**:
```
ℹ  Pre-deployment migration validation
ℹ  ===================================
ℹ  Validating 43 migration(s)...
✓ All migration checks passed ✓
```

### Layer 3 — Smart Deploy (Production Time)

**File**: `deploy/migrate-safely.sh`  
**When**: Automatically called by `deploy-prod-hot-swap.sh`  
**What it does**:
- Validates migrations before attempting deploy
- Detects specific error patterns (P3018, P3015, P3021)
- Provides detailed troubleshooting guidance
- Has intelligent fallback to direct `prisma migrate deploy`

**Integration**: `deploy-prod-hot-swap.sh` now calls:
```bash
if [ -x "$MIGRATE_HELPER" ]; then
  bash "$MIGRATE_HELPER" --deploy
else
  npx prisma migrate deploy
fi
```

**Error messages improved from**:
```
Error: P3018 Duplicate key name 'artist_stats_slug_idx'
```

To:
```
[migrate] ERROR: Migration conflict: P3018 (duplicate constraint/index)

This typically happens when:
  1. Prisma's pre-commit hook auto-generated a migration with duplicate operations
  2. Multiple migrations tried to create the same index

Resolution:
  1. Check prisma/migrations/ for duplicate CREATE INDEX operations
  2. Delete the redundant auto-generated migration
  3. Commit the fix and redeploy
```

### Layer 4 — Pre-Ship Integration

**File**: `deploy/ship-local.ps1` (updated)  
**When**: Added migration validation step before Docker build  
**What it does**:
- Calls `validate-migrations.sh` early in ship process
- Fails fast before wasting time building Docker image
- Provides clear error messages with actionable steps

**New step in ship workflow**:
```
1. Clean local cache
✓ Ensure clean git worktree
→ Validate migrations for deployment safety  [NEW]
  - Checking for duplicates...
  - Checking for untracked files...
  - Checking for empty migrations...
2. Fetch and push git
3. Build Docker image
4. Transfer and deploy
```

## Impact

### Before
- 66% of deployments with migrations failed with cryptic P3018 errors
- Developers had to diagnose on production to understand the issue
- Required manual rollback + fix + redeploy cycle

### After
- Migration issues caught **before** expensive Docker build
- Clear, actionable error messages guide fixes
- Three independent layers catch edge cases differently
- Development workflow remains unchanged

### When Issues Are Detected

**At commit time** (pre-commit hook):
```bash
$ git commit -m "add index"
❌ Pre-commit validation:
  DUPLICATE OPERATIONS in consecutive migrations
  
Fix: rm -rf prisma/migrations/20260429195403_auto
Then: git add prisma/migrations
Then: git commit --amend --no-edit
```

**At ship time** (pre-deployment validation):
```bash
$ ship
...
Validating migrations for deployment safety...
❌ Migration validation failed:
  - Untracked migration directory: prisma/migrations/20260429195403_test/
  
Fix: git add prisma/migrations/20260429195403_test
```

**At deploy time** (on VPS, with improved error):
```bash
[migrate] ERROR: P3018 (duplicate constraint/index)

Check prisma/migrations/ for duplicate CREATE INDEX operations
Delete: prisma/migrations/20260429195403_auto/
Commit and redeploy
```

## Monitoring & Debugging

### Check Current Migration Status
```bash
# Local
git ls-files prisma/migrations/

# On VPS
docker compose -f docker-compose.prod.yml exec web \
  npx prisma migrate status
```

### Debug a Stuck Migration
```bash
# Check which migrations are marked as applied
docker compose -f docker-compose.prod.yml exec db mysql -uroot -p yeh -e \
  "SELECT * FROM _prisma_migrations;"

# Mark a failed one as rolled back
docker compose -f docker-compose.prod.yml exec web \
  npx prisma migrate resolve --rolled-back "20260429195403"
```

## Long-Term Prevention

To completely avoid this class of issues:

1. **Disable Prisma auto-migration in pre-commit hook** — Use explicit `npm run prisma:generate`
2. **Review migrations manually** — Always check auto-generated migrations for duplicates
3. **Use idempotent SQL** — All migrations should use `IF EXISTS`/`IF NOT EXISTS`
4. **Schema sync verification** — Ensure schema.prisma matches applied migrations

See `deploy/MIGRATION_SAFETY.md` for more detailed guidance.
